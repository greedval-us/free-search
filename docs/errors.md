# Error Handling Guide

Этот документ фиксирует, какие исключения и ошибки использовать в проекте, чтобы не смешивать технические детали, бизнес-валидацию и пользовательские ответы.

## Цели

- Не показывать пользователю технические сообщения и внутренние детали инфраструктуры.
- Держать бизнес-ошибки в доменном слое, а HTTP-представление ошибок в одном месте.
- Сделать код предсказуемым: по типу исключения должно быть понятно, что именно произошло.

## Короткое правило

- `app/Exceptions/Public/*`:
  использовать для ошибок, которые можно безопасно отдать пользователю/API.
- `app/Exceptions/Domain/*`:
  использовать для внутренних бизнес-ошибок и инвариантов доменной модели.
- `Illuminate\Validation\ValidationException`:
  использовать в HTTP-слое, когда нужно привязать ошибку к конкретному полю формы.
- Не использовать `RuntimeException` и `InvalidArgumentException` для новых сценариев, если ошибка относится к одному из случаев выше.

## Public Exceptions

Папка: `app/Exceptions/Public`

Используются, когда ошибка должна:

- попасть в JSON-ответ;
- пройти через глобальный handler в `bootstrap/app.php`;
- иметь контролируемый `status`, `code` и локализуемое сообщение.

Текущие типы:

- `IntegrationMisconfiguredException`
  для неверной или отсутствующей конфигурации интеграции.
- `ExternalServiceUnavailableException`
  для сетевой недоступности внешнего сервиса.
- `ExternalServiceRequestException`
  для неуспешного ответа внешнего API, включая `429`.
- `PublicResourceNotFoundException`
  для ситуаций, когда пользовательский ресурс не найден.
- `PublicValidationException`
  для безопасных ошибок пользовательского ввода на прикладном уровне.

Когда использовать:

- YouTube/Mastodon/Bluesky клиент вернул ошибку.
- Внешняя интеграция не настроена.
- Пользователь запросил сущность, которой нет.
- Прикладной сервис обнаружил безопасную для пользователя проблему входных данных.

Когда не использовать:

- Для ошибок, которые относятся только к внутренней модели и ещё не должны превращаться в HTTP-ответ.
- Для обычной полевой валидации в `FormRequest`.

## Domain Exceptions

Папка: `app/Exceptions/Domain`

Используются для внутренних инвариантов и бизнес-правил. Они не обязаны напрямую знать про HTTP, JSON или локализацию для API.

Текущие типы:

- `DomainValidationException`
  для некорректных данных при создании/нормализации доменных DTO и value-like объектов.

Когда использовать:

- DTO не может быть корректно собран из входных данных.
- Нарушен инвариант бизнес-объекта.
- Ошибка относится к домену, а не к транспортному слою.

Пример:

- `SearchMessagesDTO::fromArray()` не получил обязательный `peer`.
- `SearchParticipantsDTO::fromArray()` получил неподдерживаемый `filter`.

## Form Validation

Используйте `FormRequest` или `ValidationException::withMessages()` когда:

- ошибка должна быть привязана к полю формы;
- её нужно показать в стандартном Laravel/Inertia flow ошибок;
- это именно валидация пользовательского ввода на уровне HTTP.

Пример:

- `BillingController` ловит `SubscriptionActivationException` и преобразует её в ошибку поля `activation_token`.

## Domain-Specific Exceptions

Если ошибка часто встречается в одном бизнес-сценарии, лучше дать ей отдельный тип, чем таскать строковые причины по коду.

Пример:

- `SubscriptionActivationException`
  использует named constructors:
  - `invalid()`
  - `used()`
  - `expired()`

Правило:

- если у сценария есть устойчивый набор причин и отдельное поведение, заводим отдельное исключение;
- если это разовая техническая ситуация для API, обычно достаточно одного из `Public/*`.

## Что выбирать

1. Ошибка должна уйти пользователю через JSON API?
Используй `Public` exception.

2. Ошибка относится к внутреннему бизнес-объекту или DTO?
Используй `Domain` exception.

3. Ошибка должна подсветить конкретное поле формы?
Используй `ValidationException` или `FormRequest`.

4. Ошибка специфична для одного бизнес-процесса?
Сделай отдельное named exception по образцу `SubscriptionActivationException`.

## Чего избегать

- `throw new RuntimeException(...)` в новых прикладных и доменных сценариях.
- `throw new InvalidArgumentException(...)` для бизнес-валидации без собственного типа.
- Ловить низкоуровневую ошибку в контроллере и вручную собирать JSON в каждом endpoint.
- Передавать `$exception->getMessage()` пользователю напрямую.
- Хардкодить пользовательские тексты в сервисах, если это публичное сообщение.

## Практические примеры

- Внешний API недоступен:
  `ExternalServiceUnavailableException`
- Интеграция не настроена:
  `IntegrationMisconfiguredException`
- Канал/аккаунт/хэштег не найден:
  `PublicResourceNotFoundException`
- Некорректный параметр при создании DTO:
  `DomainValidationException`
- Ошибка токена активации в настройках биллинга:
  `SubscriptionActivationException` + `ValidationException::withMessages()`

## Правило для новых изменений

При добавлении новой ошибки проверь:

1. Это публичная ошибка API или внутренняя доменная ошибка?
2. Нужен ли ей отдельный тип или достаточно существующего?
3. Не утекает ли технический текст пользователю?
4. Есть ли тест на ошибочный сценарий, если поведение важно для пользователя?
